# Migrating HomeKonet to a New Server

This repo is deliberately structured so most of the stack needs **zero**
migration effort. This doc covers what actually needs to move, and the exact
steps to move it.

## What does NOT need migrating

| Component | Why |
|---|---|
| **Database** | Postgres runs on [Neon](https://neon.tech), not on this server. The new server just needs the same `DATABASE_URL` in `backend/.env` — no dump/restore, no downtime. |
| **Redis** | Pure cache + Celery broker + Channels layer. Empty on a fresh container is fine — it just repopulates. |
| **Static files** (`static_files` Docker volume) | Regenerated by `collectstatic` every time the `backend` container starts. |
| **Docker images, nginx config, Dockerfiles** | All in git. `git clone` + `docker compose build` reproduces them exactly. |

## What DOES need migrating

Everything gitignored that isn't disposable:

| Item | Contains |
|---|---|
| `backend/media/` | Real user-uploaded photos/attachments (~80MB and growing). The only irreplaceable data that lives on local disk. |
| `.env`, `backend/.env`, `frontend/.env` | Secrets: DB credentials, Stripe/Brevo/MTN keys, Django secret key, VAPID keys. |
| `nginx/ssl/` | The TLS cert/key. Not strictly required (you can re-issue), but copying it lets the new server serve HTTPS immediately instead of waiting on a fresh Certbot run. |
| Certbot renewal hooks | Live in `/etc/letsencrypt/` on the server, outside git. Versioned in `scripts/install-certbot-hooks.sh` so they're reproducible. |

`scripts/backup.sh` / `scripts/restore.sh` handle the first three as one encrypted archive.

## Migration steps

### 1. On the OLD server: take a backup

```bash
cd /opt/homekonet
bash scripts/backup.sh
```

You'll be prompted for a passphrase (or set `BACKUP_PASSPHRASE=... bash scripts/backup.sh` to skip the prompt) — **remember it**, you'll need it to restore. This produces `backups/homekonet-backup-<timestamp>.tar.gz.gpg`, gitignored and encrypted since it contains live secrets.

Copy it to the new server:

```bash
scp backups/homekonet-backup-*.tar.gz.gpg youruser@new-server:/tmp/
```

### 2. On the NEW server: base setup

Follow `docs/DEPLOYMENT.md` §1–4 (server hardening, Docker install) if this is a fresh box. Then:

```bash
git clone https://github.com/Easy-Tech-Solutions/realestate-booking-platform.git /opt/homekonet
cd /opt/homekonet
bash scripts/restore.sh /tmp/homekonet-backup-*.tar.gz.gpg
```

This extracts `backend/media/`, all three `.env` files, and `nginx/ssl/`, then fixes permissions (world-readable media so nginx's worker can serve it, `600` on secrets). It refuses to run if `backend/.env` already exists, to avoid clobbering a real deployment — pass `--force` if you're intentionally re-running it.

Open `backend/.env` and sanity-check anything server-specific — e.g. if this move is also an opportunity to rotate a credential, or if `DJANGO_ALLOWED_HOSTS`/domain values need to change.

### 3. Build and do a dry run

```bash
docker compose config --quiet   # validates syntax
docker compose build
```

Don't `up` yet — DNS still points at the old server, so a real Let's Encrypt issuance would fail domain validation from here.

### 4. Cut over DNS

Point `homekonet.com` and `www.homekonet.com` at the new server's public IP (A record for the apex, A or CNAME for `www`). Wait for propagation — check with:

```bash
python3 -c "import socket; print(socket.gethostbyname('homekonet.com'))"
```

until it returns the new server's IP.

### 5. TLS

If you restored `nginx/ssl/` in step 2, the site can serve HTTPS immediately with the copied cert. Either way, once DNS has propagated to this server, issue a fresh certificate here so renewal works going forward (the old server no longer has DNS pointing to it, so its Certbot can't renew anymore):

```bash
sudo apt-get install -y certbot   # if not already installed
sudo certbot certonly --standalone -d homekonet.com -d www.homekonet.com \
  --email your-email@example.com --agree-tos --non-interactive
sudo cp /etc/letsencrypt/live/homekonet.com/fullchain.pem nginx/ssl/fullchain.pem
sudo cp /etc/letsencrypt/live/homekonet.com/privkey.pem nginx/ssl/privkey.pem
sudo chown "$(id -un):$(id -gn)" nginx/ssl/*.pem
chmod 644 nginx/ssl/fullchain.pem && chmod 640 nginx/ssl/privkey.pem

bash scripts/install-certbot-hooks.sh homekonet.com
sudo certbot renew --dry-run --no-random-sleep-on-renew   # sanity check
```

(Ports 80/443 must be free — don't run this after `docker compose up` unless you stop the `frontend` container first.)

### 6. Go live

```bash
docker compose up -d
docker compose ps                                    # everything healthy?
curl -sk https://homekonet.com/api/health/            # {"status": "ok"}
```

Smoke-test in a browser: home page loads listings with images, login works, and (if `BREVO_API_KEY` is a real key) password reset actually sends an email.

### 7. Decommission the old server

Only after you've verified the new one for a bit — keep the old server (and its DNS-less Certbot) around as a fallback for a day or two before tearing it down.

---

## Rollback

If something's wrong after DNS cutover, just point DNS back at the old server — it's untouched and still fully functional. Nothing in this process is destructive to the source server.

## Restoring media without a full server migration

`scripts/backup.sh`/`restore.sh` also work as a plain backup/restore pair on the *same* server — e.g. before a risky change, or as a periodic off-box backup via cron:

```bash
# crontab -e
0 3 * * * cd /opt/homekonet && BACKUP_PASSPHRASE="$(cat /root/.backup-passphrase)" bash scripts/backup.sh >> /var/log/homekonet-backup.log 2>&1
```
