Management Portal Guide

Running the HomeKonet Management Portal

A section-by-section manual for staff and administrators using /management — what each screen does, who can see it, and how the underlying access model works.

🔑 Access Model

Three tiers can reach the Management Portal, and each gets a different scope of access.

TierHow it's grantedWhat it can do
SuperadminAccount role = superadmin, set via shell access only — never through the dashboard.Unconditional access to everything. Bypasses every permission check.
AdminAccount role = admin, granted via the Users screen by a superadmin.Broad access from the "Admin" preset role — but not the RBAC engine itself or Break-Glass elevation.
Staff officeris_staff plus one or more assigned Roles (e.g. Product Support Officer, Compliance Officer).Only what their assigned role(s) explicitly grant — enforced independently on every single page/action.
Every admin sees every nav item The sidebar does not hide sections you lack permission for. If a staff officer opens a page they can't use, it loads and shows an inline message like "You do not have Trust & Safety access" rather than being hidden from the menu. This is expected — don't read it as a bug when onboarding a new officer.

Under the hood, access is governed by a fine-grained RBAC (Role-Based Access Control) engine: named Roles hold grants on a tree of resources (e.g. listings.content, finances.escrow, trust_safety.flags) × actions (create/read/update/delete/execute). A grant on a parent resource automatically covers everything beneath it. Manage this from Roles & Permissions.

📊 Overview

The dashboard landing page shows top-line stats (total revenue, users, active listings, bookings), a breakdown of bookings by status, and a recent-bookings table. Each booking row has a Messages button that opens a read-only view of the guest/host conversation for that booking — useful when investigating a dispute.

The Properties tab here offers a quick approve/reject flow for listings pending review; for full search, suspension, and compliance tools, use Inventory & Listing Moderation instead.

👥 Users

The Users screen is the full user-management console: search, filter by role/active status, and act on individual or multiple accounts.

Creating a user

Set username, email, name, and role (user or agent only) with a password you set or one generated for you. Admin accounts can't be created here — those are provisioned via shell access.

Per-account actions

ActionWhat it does
View as (impersonate)Log in as the user to reproduce an issue. Requires a reason, and is fully audit-logged. Blocked for superadmin accounts.
Edit profile / email / passwordThree independent sections — changing the password force-logs the user out everywhere.
Manage rolesAssign or revoke RBAC roles. Assigning any role marks the account as staff.
Deactivate / ReactivateToggles account access without deleting data.
Delete (soft)Requires a reason. Deactivates and anonymizes the name/email/username while preserving booking/payment/review history.
Delete permanentlyRequires a reason and typing the username to confirm. If the account has bookings/payments/listings attached, this requires a second admin's approval (see Pending Approvals).

Bulk actions apply the same operations (reactivate, deactivate, assign/remove role, soft-delete) across a multi-selected set of accounts.

Granting the Admin role Only a superadmin can grant the admin role to another account. Superadmin accounts themselves are fully protected from edits/deletion in this UI.

🛡️ Roles & Permissions

Custom roles

Create a role with a name, slug, and description, then use the permission grid to grant create/read/update/delete/execute on each resource. Preset roles (the five legacy departments plus Admin/Superadmin) are marked with a lock icon and can't be deleted — they back long-standing behavior elsewhere in the system.

Assigning a role to a user

Look up a user by ID, view their current role assignments, and assign or revoke a role. The Superadmin role itself is filtered out of every assignable-role list everywhere in the dashboard — it's granted only via shell access.

Pending Approvals (dual-authorization)

Certain high-impact actions require a second, different admin to approve before they execute:

  • Refunds over $500 (and every Stripe refund, regardless of amount)
  • Suspending a host with more than 3 published listings
  • Permanently deleting a user with protected records (bookings/payments/listings)

Each pending item shows the requester, their reason, and the underlying payload. Approve & execute or Reject (with an optional reason). The system enforces that the approver cannot be the same admin who made the request.

Break-Glass Access

For incidents where an officer needs temporary full-admin power outside their normal role, request break-glass elevation (reason required, 1–8 hours). While active, every single request the account makes is logged for audit — not just the sensitive actions. Any admin with visibility can revoke an active session early.

🔒 Security & Audit Log

Two-factor authentication (MFA)

From the Security section, enable TOTP-based 2FA: scan a QR code, confirm a 6-digit code, and save the eight one-time backup codes you're shown (they won't be shown again). Once enabled, logging in requires that code as a second step.

MFA is opt-in, not enforced Turning on MFA is currently a personal choice for each admin/staff account — the system does not block dashboard access for accounts that haven't enabled it. Treat this as a strong recommendation to every admin/staff member, not an automatic guarantee.

Audit Log

Every sensitive action across the portal — user changes, listing approvals, suspensions, refunds, role changes, feature flag edits, and more — is recorded here with the actor, action, target, reason, and IP address. Filter by action text or target type. This is your primary tool for reconstructing "who did what, when."

🪪 KYC & Verification Review

Two independent review queues live on this page, each following the same three-stage pipeline: Product Support → Compliance → Supervisor.

Host applications

Review the applicant's name/email/phone/address alongside their headshot and ID document, then Approve (advances to the next stage) or Decline (a reason is required).

Property verifications

Review the ownership details, deed/volume number, and MOU document (if the applicant isn't the property's owner). Decide Approve, Request Correction, or Reject (notes required except on approval).

AI pre-screen score

Each item shows an AI-generated pre-screen score with a short rationale, to help you prioritize review — hover the badge for the reasoning. This score reads only the submitted text (name, address, ownership details, etc.); it does not inspect the uploaded ID, headshot, or MOU images. Treat it as a triage aid, not a substitute for actually reviewing the documents. It shows "pending / disabled" if scoring hasn't run yet or is turned off — see Platform Settings.

🚨 Fraud & AML

Fraud flags

Flags are created automatically by rule-based detectors (rapid account creation from one IP, the same card used across multiple accounts) — click Run detectors to scan on demand. Each flag shows its type, severity, the linked account (if any), and an AI risk-score badge with a rationale. Confirm as real fraud or dismiss it, with notes.

Blocked device fingerprints

Block a specific device fingerprint (with a reason) to prevent it from registering, signing in with Google, or logging in with a password on any account.

Blacklisted locations

Ban a coordinate + radius (e.g. a known problem address) from being listed again, regardless of which account tries.

🏘️ Inventory & Listing Moderation

Listing flags

Same pattern as fraud flags: rule-based detectors catch likely duplicate listings and prices far outside the norm for a city/property type. Each flag carries an AI content-score badge based on the listing's title, description, price, and amenities (not its photos).

Global inventory search

Search every listing regardless of status (draft, pending review, published, rejected, suspended), filter to flagged-only, and see each listing's open-flag count.

Suspend / restore & compliance

Suspend a listing with a required reason, or restore a suspended one. The Compliance dialog sets a local registration number and an occupancy cap (which can't be set below the listing's current max-guest count). Bulk suspend/restore is available for multi-selected listings.

⛔ Suspensions

Issue a new suspension by user ID, type (temporary/indefinite/permanent), and reason — temporary suspensions require an end date. See stats (Total, Active Now, Revoked, Expired) and revoke an active suspension at any time with an optional note.

Dual-authorization threshold Suspending a user who owns more than 3 published listings routes through Pending Approvals instead of taking effect immediately — a second admin must approve it.

💰 Finance & Legal Center

Summary & transactions

See gross collected, total refunded, net revenue, commission revenue, and payout counts. Search/filter transactions and export to CSV.

Refunds

Mobile Money refunds are available directly on completed/partially-refunded transactions (amount + required reason). Stripe refunds have their own section — every Stripe refund requires a second admin's approval, regardless of amount; Mobile Money refunds only require it above $500.

Escrow

Hold a booking's payment (with a reason) to pause disbursement while investigating a dispute or fraud flag, and release it once resolved.

Occupancy tax

Manage per-city tax rates and view a computed liability report by jurisdiction. This is a records tool only — it does not file or remit taxes automatically.

🎧 Support

See ticket stats (Open, In Progress, Pending User, Resolved, Closed, SLA Breached) and work tickets: change status/priority, assign to a teammate, reply in the thread, or escalate with notes. A separate tab handles contact-form inquiries — reply by email or mark as read.

🛟 AirCover Claims

Review claims filed for property damage, missing items, cleanliness, or safety issues. Set an approved amount and notes, then Approve or Deny.

Approving doesn't pay out Approving a claim only records the approved amount — it does not move money. Issue the actual payment from the Finance & Legal Center's refund tools, referencing the claim.

🚩 Reports

User-submitted reports on listings, users, or reviews land here (distinct from Support tickets). Set status (Under Review, Resolved, Dismissed) with notes, escalate a report, or — if it names a specific user — jump straight to Suspensions with that user and report pre-filled. Bulk status changes are available for multi-selected reports.

⚙️ Platform Settings

Fees

Set the booking fee, viewing fee, service fee percentage, and transaction fee (fixed, percentage, or range) — changes apply immediately, no deployment needed.

Feature flags

Create, toggle, or delete platform-wide switches. Three keys are actually wired to real behavior today:

KeyEffect when on
maintenance_modeMakes the API read-only for non-staff users.
new_registrations_enabledWhen turned off, blocks new signups.
ai_scoring_enabledTurns on the local AI scoring used by KYC pre-screen, fraud flags, and listing moderation (and the support chatbot). Off by default — enabling it is the only way those AI scores start populating.

Any other flag you create just records an on/off switch — it won't affect behavior until a developer wires a check for it into the code.

🩺 Platform & Engineering

This page is diagnostics-only (feature flags live in Settings, above). It shows live system health — database, Redis, Celery worker, and disk status — plus a table of scheduled background tasks with their last run time and success/failure state, and a feed of the most recent application errors, so engineering staff can spot problems without needing server/SSH access.

📜 Legal Documents

Publish new versions of the Terms of Service or Privacy Policy: set a version string, effective date, a summary of changes, and the document body (structured as sections). Publishing goes live immediately on the public site and is kept as permanent version history you can review at any time.

📝 Known Gaps

A few things worth knowing so you don't go looking for them:

  • Newsletter and testimonials have no dedicated screen in the Management Portal today — manage them via Django's built-in admin site if needed.
  • MFA is opt-in for admin/staff accounts, not enforced (see Security).
  • The Management Portal sidebar shows every section to every admin/staff account regardless of their actual permissions — a "you don't have access" message on a page means exactly that, not a bug.

🔧 Troubleshooting

Symptom → cause → fix, for issues staff have actually hit while using this portal.

I assigned the "Admin" role to someone, but their Role badge in the Users list still shows the old value (e.g. "agent")

Cause: There are two related-but-different things: the account's underlying role field (what the Role badge shows) and RBAC role assignments (what "Manage roles" grants). Assigning "Admin" through Manage roles now automatically promotes the account's role field too, so the badge updates immediately — but if you're on a build from before this was fixed, the permission grant would silently take effect while the badge stayed stale, making it look like nothing happened.

Fix: Re-open the Users list (a hard refresh clears any stale cached page) and confirm the badge now reads "admin". If it still doesn't, promote directly instead: open the account's Edit profile / email / password dialog and set Role to Admin from there — this always keeps the badge and the underlying permissions in sync, both ways (demoting away from Admin removes the grant too).

I assigned the "Superadmin" role from Roles & Permissions (or Manage roles), got a success message, but the account still isn't a superadmin

Cause: The Superadmin entry under Roles & Permissions is reference-only — it exists so you can see the full permission set a superadmin has, not so it can be handed out. Real superadmin status bypasses the RBAC engine entirely (see Access Model), so assigning that entry as a role grants nothing.

Fix: The Superadmin role can no longer be selected in any assignment picker (single or bulk) — attempting it now returns a clear error instead of a false "success". There is no dashboard control for granting real superadmin status by design; it requires direct shell/Django admin access to the account's role field.

As an admin, I can't change another user's email or reset their password

Cause: Changing another account's email/password requires either superadmin, or a role granting users.pii (update) specifically — the built-in Admin preset role includes this, but a narrowly-scoped custom role might not.

Fix: Check the acting account's grants under Roles & Permissions. If they hold the "Admin" role this already works — if you saw this fail before promoting them, that was very likely the "Role badge didn't update" issue above (the promotion hadn't actually gone through yet), not a separate bug. For a custom role, add users.pii → update explicitly.

Trying to edit, deactivate, or delete an account does nothing / returns an error

Cause: Superadmin accounts are deliberately protected — they can't be edited, deactivated, soft-deleted, or permanently deleted from this dashboard at all, even by another superadmin.

Fix: This is expected, not a bug. Those changes have to be made via shell/Django admin access to that account directly.

Bulk-assigning a role to several users shows some as "failed"

Cause: Bulk actions report per-account results rather than failing the whole batch. Common reasons a single account fails: it's a superadmin account (protected), the role being assigned is Superadmin (never assignable), or the account no longer exists.

Fix: Check the per-account error in the result toast/response — the succeeded accounts are already done, so you only need to address the failed ones individually.
