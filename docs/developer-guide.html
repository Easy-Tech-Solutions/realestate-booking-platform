🏗️ Architecture Overview

Django 5 + DRF + Channels + Celery backend, React + TypeScript + Vite frontend, deployed as six Docker Compose services behind a single Nginx container. No Kubernetes, no separate load balancer, no CDN — everything in this diagram runs on one host.

Request path — a browser hitting /api/...

  1. Browser resolves homekonet.com and opens a TLS connection on 443. Anything on port 80 gets a 301 to https, except /.well-known/acme-challenge/ (Let's Encrypt HTTP-01).
  2. The frontend container's Nginx terminates TLS (cert/key bind-mounted read-only from nginx/ssl/), attaches security headers (HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy) to every response, and matches the request path against its location blocks.
  3. For /api/* or /ws/*, Nginx proxies to http://backend:8000 over the internal Compose network (service-name DNS — backend is never reachable from outside the Docker host). /api/ gets a 60s read timeout; /ws/ gets 86400s (24h) plus Upgrade/Connection headers for the WebSocket handshake.
  4. Daphne (the ASGI server) routes HTTP requests into Django/DRF as normal. CORS, security-headers, session, auth, request-logging, suspension-check, maintenance-mode, break-glass-audit, and view-tracking middleware all run in a fixed order (see Request Flow & Patterns) before the view function executes.
  5. The view authenticates via JWT (short-lived access token in the Authorization header), checks RBAC/department permissions, does its work against Postgres (and Redis cache, if applicable), and returns JSON. Everything sensitive is captured to AdminAuditLog and/or the rotating log files along the way.
  6. For anything that isn't /api/, /ws/, /admin/, /static/, or /media/, Nginx's catch-all serves the SPA's index.html (try_files $uri $uri/ /index.html) — client-side routing (React Router) takes over from there. index.html itself is served with Cache-Control: no-cache so a new deploy is picked up on next load, while hashed /assets/* chunks are cached for a year.

Every container built from Dockerfile.backend (backend, celery, celery-beat, celery-ai) shares docker-entrypoint.sh as its ENTRYPOINT: it runs migrate --no-input, then collectstatic --no-input, then execs whatever CMD/command was set. That means migrations and static collection happen redundantly on all four container starts — harmless, since both commands are idempotent, but worth knowing so a slow redeploy of, say, celery-beat alone isn't surprising.

Data & caching layer

Postgres resolves in three tiers, first one configured wins: a single DATABASE_URL (the managed-Neon path used in production) → discrete POSTGRES_* vars with DB_ENGINE=postgres (local dev against a real Postgres, sslmode=require by default) → SQLite at db.sqlite3 as a last-resort fallback for a bare checkout with no DB config at all.

Redis does three unrelated jobs in this deployment, all off one instance:

  • Celery broker + result backend — always, whenever REDIS_URL is set.
  • Django cache — only if REDIS_URL is set; otherwise falls back to a database-backed cache table (not LocMemCache), deliberately, because per-process in-memory cache can't be shared across multiple Daphne/Gunicorn workers — and this cache is what backs API rate limiting, which only works correctly if every worker sees the same counters.
  • Channels layer — Redis-backed (channels_redis) when configured, else InMemoryChannelLayer, which only fans out WebSocket messages within a single process — fine for one-off local dev, not representative of production WS behavior.
A rediss:// URL from a managed provider If REDIS_URL uses rediss:// (TLS) without an explicit ssl_cert_reqs query param, settings.py appends ssl_cert_reqs=none automatically — many managed Redis providers hand out certificates that don't chain to a publicly-trusted CA, and Python's default strict verification would otherwise refuse to connect.

Django configuration highlights

Middleware order (top to bottom, and order matters): CORS → custom security headers → Django's own SecurityMiddleware → WhiteNoise (must come right after SecurityMiddleware per its own docs) → sessions → common → CSRF → auth → request-logging (deliberately after auth, so request.user is populated when a request gets logged) → messages → clickjacking → suspension-check → maintenance-mode → break-glass audit → listing view-tracking.

Auth model: stateless JWT access tokens (15 min lifetime) carried in the Authorization header; the refresh token (14 days, rotates on every use, old one blacklisted) lives only in an httpOnly, SameSite, secure cookie scoped to /api/auth/ — JavaScript can never read it, which is the actual XSS-token-theft mitigation, not an incidental detail.

ASGI, not WSGI, in production: both wsgi.py and asgi.py exist, but only the ASGI app is actually served (via Daphne) — WSGI is there for compatibility/management-command contexts only. asgi.py builds a ProtocolTypeRouter with a plain Django app for http and a Channels URLRouter (over messaging + notifications routing) for websocket — wrapped in AllowedHostsOriginValidator in production only, so a WS handshake with a forged Origin header is rejected before it ever reaches a consumer.

CORS is allow-listed by exact origin only (no regex origins), always including the production Vercel preview domains plus whatever CORS_ALLOWED_ORIGINS/FRONTEND_ORIGIN env vars supply, with credentials enabled — meaning cookies/auth headers are only ever sent to origins explicitly on that list, never to an arbitrary subdomain.

No error-tracking SaaS (no Sentry or equivalent) — production error visibility is entirely the custom rotating JSON log files plus the platformops log-viewer/recent-errors/system-health endpoints (see Monitoring & Logging).

Frontend build & runtime

Vite + @vitejs/plugin-react + Tailwind. vite.config.ts manually splits vendor chunks by dependency (map-vendor for Leaflet, charts-vendor for Recharts/d3, query-vendor for React Query, calendar-vendor, motion-vendor, ui-vendor, router-vendor) so a change to one area doesn't invalidate the browser cache for everything else.

State/data: Zustand (src/store/appStore.ts) for auth/app state, TanStack Query for server-state caching, react-hook-form for forms. UI is a mix of hand-rolled Radix/shadcn components (most of the app) and MUI (present in package.json, used in a minority of older screens) — prefer the existing Radix/shadcn primitives in components/ui/ for anything new. Routing is React Router v7.

Real-time: no Socket.IO — useChatSocket/useNotificationSocket hooks open native browser WebSocket connections directly against VITE_WS_URL + /ws/chat/<id>/ / /ws/notifications/, matching Channels' routing on the backend exactly — no client library translation layer in between.

The built output (dist/) is copied into the frontend Nginx image at build time — there's no separate static-hosting service (no Vercel/Netlify/S3+CloudFront in this deployment); the single Nginx container is both the CDN-equivalent and the API gateway.

🧩 Request Flow & Patterns

Standard endpoint shape

Almost every endpoint is a DRF function-based view, not a viewset:

@api_view(["GET", "POST"])
@permission_classes([IsAuthenticated])
def bookings_collection(request):
    ...

Business logic that doesn't belong in a view lives in a per-app services.py — e.g. bookings/services.py is the single place booking-lifecycle transitions happen (confirm, decline, expire), called identically from views, Django admin actions, and Celery beat tasks. payments/services.py and propertyverifications/services.py follow the same convention. Serializers validate/shape input; models stay thin.

Booking state machine

pending_host → (host confirms within 7 days) → awaiting_payment → (guest pays within 10 days) → payment_received → (admin confirms) → confirmed (+ Payout created). Missed deadlines transition to expired_unconfirmed/expired_unpaid via Celery beat. Confirming a reservation auto-declines other overlapping pending_host requests on the same listing.

Permission checks

Two systems, both still live: the legacy department gate (superadmin.permissions.require_department/is_superadmin_staff) and the newer fine-grained RBAC engine (rbac.permissions.has_permission/has_any_permission/has_role). The legacy department helpers now delegate internally to the RBAC engine — see RBAC Model.

from rbac.permissions import is_full_admin, has_any_permission
if not (is_full_admin(user) or has_any_permission(user, 'listings.content')):
    return Response({'error': '...'}, status=403)

Admin audit logging

Any sensitive admin action calls superadmin.permissions.log_admin_action(request, action, target=obj, reason=..., **metadata), appending an immutable AdminAuditLog row. rbac.middleware.BreakGlassAuditMiddleware additionally logs every request made during an active break-glass session.

JWT auth flow

  • POST /api/auth/register/ — gated by the new_registrations_enabled flag and blocked device fingerprints; sends a verification email.
  • POST /api/auth/login/ — if the account has a confirmed MFA device, returns {"mfa_required": true, "mfa_token": ...} instead of tokens; otherwise issues tokens.
  • The access token is returned in the JSON body. The refresh token is set ONLY as an httpOnly cookie (AUTH_REFRESH_COOKIE_NAME="refresh_token", path /api/auth/, 14-day max age) — an explicit XSS-hardening measure, never returned in the body.
  • POST /api/auth/refresh-token/ — rotates the refresh token (ROTATE_REFRESH_TOKENS=True, BLACKLIST_AFTER_ROTATION=True); a replayed, already-rotated token raises TokenError → 401, giving reuse detection "for free."
  • POST /api/auth/google/ — verifies the Google ID token's issuer/audience/email_verified; blocks admin/staff accounts from Google sign-in (password only); no auto-linking if the email is already registered locally.

ACCESS_TOKEN_LIFETIME=15min, REFRESH_TOKEN_LIFETIME=14 days (settings.py).

⚙️ Async & Background Work

Celery setup

realestate_backend/celery.py pulls every CELERY_* setting from Django settings and auto-discovers tasks.py in every installed app. Broker/result backend is Redis. CELERY_TASK_ALWAYS_EAGER defaults to DEBUG — in a fresh dev checkout with no broker configured, tasks just run synchronously in-process.

Beat schedule

CELERY_BEAT_SCHEDULE = {
    'expire-unconfirmed-reservations': {  # hourly :00
        'task': 'bookings.tasks.expire_unconfirmed_reservations',
    },
    'expire-unpaid-reservations': {       # hourly :05
        'task': 'bookings.tasks.expire_unpaid_reservations',
    },
    'expire-suspensions': {               # hourly :10
        'task': 'suspensions.tasks.expire_suspensions',
    },
    'collect-server-metrics': {           # every 5 minutes
        'task': 'platformops.tasks.collect_server_metrics',
    },
}

The last one writes a ServerMetricSnapshot row (CPU/memory/disk/network, via psutil) and prunes anything older than 7 days — it's the write side of the history graph behind GET /api/platform-ops/server-metrics/ (see Architecture Overview and Monitoring & Logging).

The dedicated ai_scoring queue

Every aiscoring and chatbot task is declared @shared_task(queue='ai_scoring') and is consumed only by the celery-ai service (--concurrency=1 --pool=solo -Q ai_scoring) — deliberately, so only one process ever loads the ~3 GB local model. The default celery/celery-beat services don't listen on this queue; if celery-ai isn't running, those tasks just sit unconsumed.

Channels / WebSocket

realestate_backend/asgi.py splits HTTP (normal Django) from WebSocket (Channels URLRouter over messaging + notifications routing). Channel layer is Redis-backed when REDIS_URL is set, else in-memory (only works within a single process — Redis is required for realistic multi-worker behavior). Chat clients authenticate after connecting by sending {"type": "authenticate", "token": "..."}, not via the connection handshake.

🤖 Local LLM Subsystem

Qwen2.5-3B-Instruct, GGUF-quantized, CPU inference via llama-cpp-python. Powers KYC pre-screen, fraud/listing scoring, and the support chatbot — all gated by one feature flag.

aiscoring/model_service.py — lazy singleton

def get_model():
    global _model
    if _model is not None:
        return _model
    with _lock:
        if _model is not None:
            return _model
        if not os.path.exists(model_path):
            raise ModelUnavailable(...)
        from llama_cpp import Llama
        _model = Llama(model_path=model_path, n_ctx=settings.AI_MODEL_CONTEXT_SIZE,
                        n_threads=os.cpu_count() or 4, verbose=False)
        return _model
Never call this from a view get_model() is only ever called from Celery tasks (aiscoring.tasks / chatbot.tasks), never at Django startup or in the HTTP request path — so the model loads once per Celery worker process, never per web worker.

Scoring & the JSON contract

aiscoring/scorer.py prompts the model to reply with only a JSON object ({"score": 0-100, "rationale": "..."}), using create_chat_completion(..., response_format={"type": "json_object"}) for constrained decoding, with a regex fallback (\{.*\}) if strict parsing fails. Four scoring entry points — fraud flags, listing flags, host applications, property verifications — each build a domain prompt from real structured fields only; none of them do vision/OCR on uploaded documents or photos.

Task gating & observability

Each of the four aiscoring.tasks functions: checks platformops.utils.is_feature_enabled('ai_scoring_enabled', default=False) and short-circuits if off → loads the row → scores it → writes ai_score/ai_rationale back → records a TaskHeartbeat (success/failure) → on ModelUnavailable (model not downloaded), logs a warning and swallows; on any other exception, logs and re-raises.

Downloading the model

python manage.py download_ai_model         # skip if already present + checksum matches
python manage.py download_ai_model --force # force re-download

Streams to a .part temp file, verifies SHA-256 against AI_MODEL_SHA256, and only then atomically os.replace()s it into place — never leaves a corrupt file at the real path. This command runs automatically as the first step of the celery-ai container's start command, so the very first boot downloads the ~3 GB file before the worker becomes ready.

Chatbot reuse

chatbot/responder.py imports get_model directly from aiscoring.model_service — same process-wide singleton, so as long as one celery-ai process handles both, the model loads once and is shared. The chatbot builds a system prompt from a knowledge base (chatbot/knowledge_base.py — static FAQ/product copy plus a 60-second-cached query of published listings) and a token-budgeted conversation history, constraining output to {"reply": str, "needs_agent": bool}. POST /api/chatbot/handoff/ converts a transcript into a real support.SupportTicket.

Feature-flag default is off ai_scoring_enabled defaults to False so a fresh deploy never loads the model until an admin opts in from Management Portal → Settings. Both aiscoring and chatbot check this flag independently before doing any inference.

🗄️ Database & Migrations

Engine selection priority in settings.py: DATABASE_URL (managed Postgres, e.g. Neon) → discrete DB_ENGINE=postgres/POSTGRES_* vars → local SQLite fallback for a bare checkout with no DB config.

Running migrations

# Local dev
cd backend
python manage.py migrate
python manage.py createcachetable   # only needed if not using Redis cache

# Docker — automatic
# docker-entrypoint.sh runs `migrate --no-input` then `collectstatic --no-input`
# before exec'ing the real process, on every container start
# (backend, celery, celery-beat, celery-ai all do this redundantly — harmless, idempotent)

Some migrations are data migrations, not just schema — e.g. hostapplications/migrations/0002_create_reviewer_groups.py seeds reviewer Django Groups; listings/migrations/0011_... and 0018_... seed/rename property categories. Be careful squashing or deleting these.

Before considering a deploy release-ready, confirm zero pending migrations: python manage.py showmigrations (the release-check script enforces this automatically — see Testing & Release Gate).

🚀 Deployment & Infrastructure

Images

Dockerfile.backend is a two-stage build. The builder stage installs gcc g++ cmake libpq-dev libjpeg-dev zlib1g-dev and runs pip install -r requirements.txt.

llama-cpp-python compiles from source There's no prebuilt wheel path assumed here — llama-cpp-python compiles its C++ backend at install time, making backend image builds noticeably slower whenever that dependency or the base image cache is invalidated. Budget extra time for a clean docker compose build backend.

Dockerfile.frontend uses node:20-alpine to build the Vite app, then nginx:1.27-alpine to serve dist/.

VITE_* vars are baked at build time Changing VITE_API_URL/VITE_WS_URL/etc. always requires rebuilding the frontend image — restarting the container alone won't pick up new values.

Deploying an update

cd /opt/homekonet
git pull origin main
docker compose build
docker compose up -d --force-recreate
docker compose ps
docker compose logs --tail=50 backend

Migrations and collectstatic happen automatically via the entrypoint on every recreated container — no separate manual migrate step for a Compose deploy. To update a single service: docker compose build backend && docker compose up -d --no-deps --force-recreate backend.

Nginx

nginx/nginx.conf terminates TLS, sets security headers (HSTS, X-Frame-Options, etc.), and proxies: /api/ (60s read timeout) and /ws/ (86400s read timeout, for long-lived WebSockets) to backend:8000; serves /static/, /media/, and /assets/ with long cache lifetimes; falls back to the SPA's index.html for everything else.

⌨️ CLI Reference

Every command actually needed to run, deploy, debug, and maintain this app on the server (/opt/homekonet), grouped by task. All six services (redis, backend, celery, celery-beat, celery-ai, frontend) run under Docker Compose — there is no separate systemd unit for the app itself, only for the docker daemon.

Container lifecycle

# Build one service (rebuild after any code change before deploying it)
docker compose build backend
docker compose build frontend
docker compose build                        # every service

# Deploy: recreate container(s) from the freshly built image
docker compose up -d --force-recreate backend celery celery-beat
docker compose up -d --force-recreate frontend
docker compose up -d --force-recreate       # everything

# Update a single service without touching the others
docker compose build backend && docker compose up -d --no-deps --force-recreate backend

# Status of every service (state, health, ports)
docker compose ps

# Stop / start without rebuilding or recreating
docker compose stop backend
docker compose start backend
docker compose restart backend              # stop+start, keeps the existing image

# Tear everything down (containers only — named volumes survive by default)
docker compose down
# Add --volumes only if you explicitly intend to wipe redis_data/static_files/ai_models — this is destructive.
The entrypoint always runs migrate + collectstatic first Every container start (up, restart, or a fresh run) executes docker-entrypoint.sh, which runs migrate --no-input and collectstatic --no-input before the real process starts — including before a one-off docker compose run command you pass on the same line. This is why a migration file created via makemigrations inside a run never applies in that same invocation — see Common Errors #5.

Django management commands

Two ways to run one: exec (inside an already-running container) or run --rm (spins up a fresh, throwaway container). Prefer exec for anything read-only or quick; run --rm when the target container isn't up yet or you want full isolation.

# Interactive Django shell against the real database
docker compose exec backend python manage.py shell
docker compose exec -T backend python manage.py shell < script.py   # -T: pipe a script in non-interactively

# Check for pending migrations / config problems without changing anything
docker compose run --rm backend python manage.py check
docker compose exec backend python manage.py showmigrations

# Apply migrations by hand (normally automatic via the entrypoint — see the callout above)
docker compose exec backend python manage.py migrate

# Create a new migration — the container is ephemeral, so bind-mount the migrations
# folder back to the host or the generated file vanishes when the container exits:
docker compose run --rm \
  -v $(pwd)/backend/<app>/migrations:/app/<app>/migrations \
  backend python manage.py makemigrations <app>

# Provision a full admin/superuser account directly (the only supported way to grant
# role='superadmin' — there is intentionally no dashboard control for this)
docker compose exec backend python manage.py createsuperuser

# Raw SQL shell against the real (Neon/Postgres) database
docker compose exec backend python manage.py dbshell

# Any other one-off management command follows the same pattern:
docker compose exec backend python manage.py <command> [args]

Shell access inside a container

docker compose exec backend bash            # or sh, if bash isn't installed in that image

# One-off Python (not manage.py shell — no Django app registry loaded)

🛠️ Common Dev Tasks

Add a new Django app/endpoint

  1. python manage.py startapp <name>, add it to INSTALLED_APPS.
  2. Follow the convention: models.py → serializers.py → services.py (business logic) → views.py (thin @api_view functions) → urls.py → register under api/<name>/ in realestate_backend/urls.py.
  3. For admin-gated endpoints, check against an existing or new node in rbac/resources.py's resource tree, and call log_admin_action for anything sensitive.
  4. python manage.py makemigrations <name> && python manage.py migrate.

Add a new Celery task

Regular job → @shared_task in the app's tasks.py (auto-discovered, runs on the default queue); register a schedule in CELERY_BEAT_SCHEDULE if it's periodic. LLM-backed job → put it in (or alongside) aiscoring/tasks.py, decorate @shared_task(queue='ai_scoring'), gate it behind is_feature_enabled('ai_scoring_enabled', default=False), and call TaskHeartbeat.record(...) at the end.

Add a new feature flag

Creating a FeatureFlag row alone does nothing — it's just a registered switch. You must also add an is_feature_enabled('your_key', default=...) check at the actual enforcement point in code.

🔧 Troubleshooting

A Celery task is failing

  1. Check docker compose logs celery|celery-beat|celery-ai for the traceback.
  2. Check the TaskHeartbeat row for that task name via /api/platform-ops/system-health/ — last_error holds the last failure message.
  3. Check /api/platform-ops/recent-errors/ for the errors.log tail.
  4. For the ai_scoring queue specifically: confirm ai_scoring_enabled is on, and that the GGUF file actually exists at AI_MODEL_PATH (re-run download_ai_model, or --force if the checksum is bad).

Rebuilding after a dependency change

Python deps (backend/requirements.txt): docker compose build backend celery celery-beat celery-ai (they all share Dockerfile.backend) — expect this to be slow if llama-cpp-python or the base image cache was invalidated. Node deps (frontend/package.json): docker compose build frontend.

Migrations out of sync

Standard Django approach: python manage.py migrate <app> zero to unapply, remove migration files, makemigrations again — but watch for the data migrations called out in Database & Migrations; squashing/deleting them needs care since they seed real data, not just schema.

Connecting to the database

Since there's no local db container, connect directly to whatever DATABASE_URL/POSTGRES_* point to — python manage.py dbshell, or psql directly with the credentials in backend/.env.

🐞 Common Errors — Real Incidents

Actual bugs hit and fixed in this codebase, kept here so the same class of mistake is faster to recognize next time. Each entry is symptom → root cause → fix → how to avoid it again.

1. "Failed to fetch" on login/register, no server-side error at all

Symptom: the browser shows a generic "Failed to fetch" toast; DevTools console shows something like:

Access to fetch at 'https://homekonet.com/api/auth/login/' from origin 'https://www.homekonet.com'
has been blocked by CORS policy: Request header field x-device-fingerprint is not allowed
by Access-Control-Allow-Headers in preflight response.

Cause: django-cors-headers only allows a small default header set unless CORS_ALLOW_HEADERS is set explicitly. The frontend had started sending a custom X-Device-Fingerprint header on login, but the backend's CORS config was never updated to allow it — the browser blocked the request in its preflight check before it ever reached Django. This only shows up when the frontend and backend are on different origins (here, www.homekonet.com calling homekonet.com); same-origin requests never preflight, so this can look fine in a dev setup where both are served from one origin.

Fix (realestate_backend/settings.py):

from corsheaders.defaults import default_headers as _cors_default_headers
CORS_ALLOW_HEADERS = [*_cors_default_headers, "x-device-fingerprint"]

Avoid it again: whenever you add a new custom request header anywhere in the frontend, immediately check/update CORS_ALLOW_HEADERS. Verify with a real preflight, not a same-origin curl call:

curl -X OPTIONS https://homekonet.com/api/auth/login/ \
  -H "Origin: https://www.homekonet.com" \
  -H "Access-Control-Request-Method: POST" \
  -H "Access-Control-Request-Headers: content-type,x-device-fingerprint" -D -

2. A duplicate function/property name silently shadows the correct implementation

Symptom: a feature (account deletion) calls the wrong implementation with no obvious error at the call site — a 405 or 500 despite the "right" code looking correct wherever you look for it.

Cause — found twice, one in each language:

  • frontend/src/services/api/users.ts defined deleteAccount twice in the same object literal (once DELETE, once POST). JS object literals keep only the last duplicate key — the correct DELETE version was silently discarded at runtime.
  • backend/users/views.py had from .deletion import delete_account at the top, then defined its own unrelated, never-wired-to-any-URL view function also named delete_account(request) later in the same file. The later def rebinds the module-level name — every caller after that point silently got the wrong function (which expected an HttpRequest, not a User, and crashed with a 500).

Fix: delete the incorrect/dead duplicate in each case; keep the one that's actually wired up.

Avoid it again: npm run typecheck does catch the JS case — TS1117: An object literal cannot have multiple properties with the same name — make typecheck part of your pre-merge checklist rather than eyeballing diffs. Python has no built-in check for this; a linter rule like Ruff's/Pyflakes' F811 ("redefinition of unused name") catches it and is worth adding to the pre-commit hooks (only Bandit runs today). Until then, grep before adding a new definition of an existing name: grep -rn "def delete_account\|delete_account =" backend/.

3. NameError after refactoring out a shared helper function

Symptom: an endpoint that previously worked (MFA login step-up) suddenly 500s with NameError: name '_verify_code_or_backup' is not defined — even though manage.py check reported zero issues right before deploying.

Cause: a private module-level helper was moved onto a model method (MFADevice.verify_code_or_backup()), and one of its two call sites was updated while the other (mfa_verify_login) was missed. manage.py check only validates configuration/imports at load time — it never executes view function bodies, so a stale reference inside a function is invisible until that exact code path actually runs.

Fix: grepped for the old name, found the missed call site, updated it to device.verify_code_or_backup(code).

Avoid it again: after renaming/moving/removing any function, grep -rn <old_name> backend/ across the whole repo — don't rely on manage.py check or a partial read of one file. Then actually exercise the code path (or write a test) rather than trusting a clean-looking diff.

4. Docker build fails immediately with a BuildKit RPC error

Symptom: docker compose build fails before any real build step runs:

target backend: failed to receive status: rpc error: code = Unavailable desc = error reading from server: EOF

Cause: a transient BuildKit daemon connection hiccup — not a real problem with the Dockerfile, cache, or dependencies.

Fix: just retry docker compose build. It succeeded immediately on the next attempt with the same warm cache.

5. A migration you just wrote doesn't apply on the same docker compose run

Symptom: you run docker compose run --rm backend python manage.py makemigrations, the new migration file is created — but the database still doesn't have the new column/table.

Cause: docker-entrypoint.sh (the container's ENTRYPOINT) runs migrate --no-input automatically before execing your command. On that run, migrate finds nothing pending (the migration file doesn't exist yet) — only afterward does your makemigrations command create it, one step too late for that same container start to apply it.

Fix: run makemigrations, then a separate docker compose run --rm backend python manage.py migrate (or restart any service normally — the next entrypoint run picks up the new file automatically).

6. Testing a DRF view directly in a script silently gets AnonymousUser

Symptom: calling a view function from a Django-shell script with django.test.RequestFactory and manually setting request.user = some_user still returns 403, as if the user were anonymous.

Cause: @api_view-decorated views re-wrap the raw request into a DRF Request during dispatch(), which resolves .user through DRF's own authentication classes — it ignores whatever you set on the raw Django request beforehand.

Fix: use rest_framework.test.APIRequestFactory + force_authenticate(request, user=...) instead of django.test.RequestFactory:

from rest_framework.test import APIRequestFactory, force_authenticate

rf = APIRequestFactory()
req = rf.post('/api/some/endpoint/', {...}, format='json')
force_authenticate(req, user=some_user)
response = some_view(req)

7. Two independent "make this account an Admin" code paths drifted apart

Symptom: a staff member assigns the admin preset Role to a user via POST /api/rbac/user-roles/ (the Manage Roles UI). The permission grant works immediately — but the account's User.role field (what the Role badge in the Users list reads, and what is_full_admin-adjacent checks key off) stays at its old value. It looks like the assignment silently did nothing.

Cause: users.admin_views.admin_update_user (Edit Profile's role dropdown) and rbac.views.user_roles_collection (Manage Roles' role picker) are two separate endpoints that both affect "is this account Admin-tier" — but only the first one was ever taught to flip User.role. The RBAC engine and the account's tier field are related concepts (a preset Role backs a tier) but are stored independently; nothing kept them in sync when the *other* endpoint was used.

Fix (rbac/views.py, both the assign and revoke handlers):

# On assign: promote the role field when the 'admin' preset Role is granted
if role.slug == 'admin' and target_user.role not in ('admin', 'superadmin'):
    target_user.role = 'admin'
    update_fields.append('role')

# On revoke: demote it back when that same Role is removed
if assignment.role.slug == 'admin' and target_user.role == 'admin':
    target_user.role = 'user'
    target_user.save(update_fields=['role'])

The same two-branch logic was duplicated into users.admin_views.admin_bulk_user_action's assign_role/remove_role cases, since that endpoint reimplements role assignment directly against the ORM rather than calling the RBAC view.

Avoid it again: whenever a concept is representable through more than one endpoint/table (here: "Admin-tier" via both a plain model field and an RBAC Role), grep for every write path to that concept before shipping either one — grep -rn "UserRoleAssignment.objects.create\|UserRoleAssignment.objects.get_or_create" backend/ — and keep them in sync at the same layer (the view), not by hoping callers remember to do both.

8. A "documentation-only" preset Role was fully assignable through the normal UI

Symptom: a staff member assigns the superadmin preset Role to another account via the normal role picker. The request returns 200/201 (a UserRoleAssignment row really is created) — but the target account gains no actual capability, since real superadmin status comes from User.role == 'superadmin' / is_superuser, which rbac.permissions.is_full_admin checks directly and which bypasses the RBAC engine before any RolePermission row is ever consulted (has_permission/has_any_permission both short-circuit on is_full_admin(user) first). A visible "success" response with zero effect is worse than an error — nothing told the caller their action was a no-op.

Cause: the superadmin Role was seeded into rbac.preset_roles.PRESET_ROLE_DEFINITIONS purely so its full permission set would be visible/documented in the Roles & Permissions screen (see RBAC Model for Developers) — but nothing on the assignment endpoint distinguished "a role that documents a bypass tier" from "a role that actually grants access."

Fix — reject the assignment outright, in every code path that creates a UserRoleAssignment (rbac.views.user_roles_collection and users.admin_views.admin_bulk_user_action's assign_role branch):

if role.slug == 'superadmin':
    return Response({'error': (
        'The Superadmin role exists here for reference only — assigning it grants no '
        'extra access ... can only be granted via shell/Django admin access.'
    )}, status=status.HTTP_400_BAD_REQUEST)

Also filtered out of every frontend assignable-role picker (AdminUsers.tsx, AdminRoles.tsx) so it can't be selected in the first place — defense in depth, since the backend block is the actual guarantee.

Avoid it again: if a database row exists purely for display/documentation (not to be acted on), make the code that would act on it refuse explicitly rather than relying on it "just doing nothing harmful" — a silent no-op that returns 200 is indistinguishable from success to both the caller and whoever's debugging their report later.

🔐 RBAC Model for Developers

Data model

  • Role — a named bundle of permissions. is_preset=True marks the five auto-created roles backing the legacy department system (trust_safety, inventory, support, finance, engineering) plus admin/superadmin.
  • RolePermission — one (resource, action) grant on a role; a grant on a parent resource path implies every descendant.
  • UserRoleAssignment — user↔role link; a user's effective permissions are the union of every assigned role's grants.
  • BreakGlassSession — temporary full-admin elevation; while active, permission checks treat the user as a full admin, and every request is captured as an audit-log entry.
  • PendingApproval — the record backing dual-authorization.

Helper functions (rbac/permissions.py)

FunctionWhat it checks
is_full_admin(user)Only role='superadmin' or Django is_superuser — the lesser 'admin' role does NOT get an automatic bypass.
has_active_break_glass(user)Any non-revoked, non-expired break-glass session.
has_role(user, slug)Full-admin/break-glass short-circuit, else checks UserRoleAssignment.
has_permission(user, resource, action)Does any assigned role grant this action on this resource or an ancestor?
has_any_permission(user, resource)Coarser — any action at all on this resource/ancestors.

Dual-authorization (rbac/dual_auth.py)

A generic four-eyes mechanism. A view registers an executor under an action key via @register_executor('some.action'), then calls submit_or_execute(action_key, payload, requested_by, reason, requires_dual_auth) — the caller computes requires_dual_auth from its own business rule (e.g. a refund-amount threshold). If true, a PendingApproval is created instead of running immediately; a different admin must call approve() (raises if the same user tries to approve their own request).

Legacy integration

superadmin.permissions still exposes the old department-based API (is_full_admin, is_superadmin_staff, user_departments, require_department) for backward compatibility, but internally it now delegates to rbac.permissions.has_role() against the matching preset-role slug.
